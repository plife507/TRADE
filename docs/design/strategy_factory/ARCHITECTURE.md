## Stage 1: Conceive — Strategy Generation

### Approach: Template + Parameter Grid

A "strategy template" is a Play YAML with placeholder parameters. A parameter
grid defines the search space. The generator produces the cartesian product
(or a sampled subset for large spaces).

```
Template (1)  ×  ParamGrid (N combinations)  =  N Play YAMLs
```

### Template Example

```yaml
# templates/ema_crossover.yml
name: "ema_cross_{ema_fast}_{ema_slow}_{exec}"
description: "EMA crossover generated by strategy factory"

symbol_universe: [BTCUSDT]

timeframes:
  low_tf: "15m"
  med_tf: "1h"
  high_tf: "D"
  exec: "{exec}"           # Parameterized

features:
  ema_fast:
    indicator: ema
    params: { length: "{ema_fast}" }  # Parameterized
    timeframe: exec
  ema_slow:
    indicator: ema
    params: { length: "{ema_slow}" }  # Parameterized
    timeframe: exec

actions:
  entry_long:
    all:
      - [ema_fast, ">", ema_slow]
      - [ema_fast.prev, "<=", ema_slow.prev]
  entry_short:
    all:
      - [ema_fast, "<", ema_slow]
      - [ema_fast.prev, ">=", ema_slow.prev]
  exit_long:
    all:
      - [ema_fast, "<", ema_slow]
  exit_short:
    all:
      - [ema_fast, ">", ema_slow]

risk:
  tp_pct: "{tp_pct}"       # Parameterized
  sl_pct: "{sl_pct}"       # Parameterized
  leverage: 1
```

### Parameter Grid

```yaml
# grids/ema_crossover_grid.yml
template: ema_crossover
params:
  ema_fast: [8, 12, 21]
  ema_slow: [50, 100, 200]
  exec: ["low_tf", "med_tf"]
  tp_pct: [1.0, 2.0, 3.0]
  sl_pct: [0.5, 1.0, 1.5]

# Total combinations: 3 × 3 × 2 × 3 × 3 = 162 plays
```

### Generation Modes

| Mode | Method | Scale |
|------|--------|-------|
| Grid | Cartesian product of all params | Exact (can be large) |
| Random | Random sample from grid | Controllable (e.g. 500 of 10,000) |
| Latin Hypercube | Space-filling sample | Better coverage than random |
| Genetic | Evolve from previous winners | Iterative refinement |

## Stage 2: Mass Backtest

### Existing Infrastructure

The parallel backtest runner (`src/engine/runners/parallel.py`) already
handles this. The key additions:

1. **Batch discovery** — point at a factory run folder, discover all plays
2. **Result collection** — aggregate results into a leaderboard
3. **Two-pass filtering** — synthetic first (fast, cheap), then real data (slower, trustworthy)

### Execution Flow

```
Factory Run (162 plays)
  │
  ├── Pass 1: Synthetic Backtest (all 162)
  │     │
  │     ├── run_backtests_parallel(play_ids, max_workers=8)
  │     ├── ~3s per play × 162 / 8 workers = ~1 min
  │     │
  │     └── Filter: Keep top 30% by Sharpe ratio
  │           Result: ~50 plays survive
  │
  └── Pass 2: Real-Data Backtest (top 50)
        │
        ├── Serial sync gate (DuckDB write)
        ├── run_backtests_parallel(survivors, max_workers=8)
        ├── ~10s per play × 50 / 8 workers = ~1 min
        │
        └── Filter: Keep top 20% by Sharpe + max drawdown
              Result: ~10 plays survive → promote to Live Sim
```

### Scoring Function

```python
def score_play(result: BacktestResult) -> float:
    """Composite score for ranking play quality."""
    sharpe = result.sharpe_ratio
    max_dd = result.max_drawdown_pct
    trade_count = result.total_trades
    win_rate = result.win_rate

    # Minimum thresholds (auto-fail)
    if trade_count < 10:
        return -999  # Not enough data
    if max_dd > 30:
        return -999  # Unacceptable drawdown
    if sharpe < 0:
        return -999  # Negative risk-adjusted return

    # Composite score (higher = better)
    score = (
        sharpe * 0.40           # Risk-adjusted return
        + win_rate * 0.20       # Consistency
        + (1 - max_dd/100) * 0.25  # Drawdown penalty
        + min(trade_count / 50, 1.0) * 0.15  # Activity (capped)
    )
    return score
```

## Stage 3: Live Sim — The Core New Piece

See [LIVE_SIM_ENGINE.md](LIVE_SIM_ENGINE.md) for detailed design.

### Key Concept

Feed real-time WebSocket market data into SimulatedExchange. Each engine
runs independently with its own simulated balance, positions, and PnL.
Zero exchange interaction — purely local computation.

```
WebSocket (1 per symbol)
  └── DataFanout
       ├── Engine 1 (Play A) → SimExchange → equity tracking
       ├── Engine 2 (Play B) → SimExchange → equity tracking
       ├── Engine 3 (Play C) → SimExchange → equity tracking
       └── ... (10-50 concurrent)
```

### Why Not Existing Shadow Mode?

| | Shadow (existing) | Live Sim (new) |
|---|---|---|
| Data source | LiveDataProvider (WebSocket) | LiveDataProvider (WebSocket) |
| Exchange | ShadowExchange (no-op) | SimulatedExchange (full sim) |
| Positions | Never created | Tracked, opened/closed |
| PnL | None | Full equity curve |
| TP/SL | Logged, never fire | Fire and fill like backtest |
| Liquidation | None | Simulated |
| Use case | "Does this signal fire?" | "Does this strategy make money?" |

Live Sim is a **forward-test** — it proves the strategy works on unseen
real-time data, not just historical data. This is the critical overfitting
filter that backtesting alone cannot provide.

## Stage 4: Promotion Pipeline

See [PROMOTION_PIPELINE.md](PROMOTION_PIPELINE.md) for detailed design.

### Promotion Criteria

```
Live Sim (7+ days, 10+ trades)
  │
  ├── Sharpe > 1.0  AND
  ├── Max DD < 15%  AND
  ├── Win rate > 40% AND
  ├── Profit factor > 1.5
  │
  └── → PROMOTE to Demo
        │
        ├── Demo (14+ days, 20+ trades)
        │   │
        │   ├── Matches live sim within 20% AND
        │   ├── No critical errors AND
        │   ├── Fill quality acceptable
        │   │
        │   └── → PROMOTE to Live (sub-account)
        │
        └── DEMOTE back to Live Sim if criteria fail
```

### Sub-Account Architecture

```
Master Account (capital pool, no trading)
  │
  ├── Sub-Account: factory_winner_001
  │     ├── Play: ema_cross_12_100_low
  │     ├── Allocated: $1,000
  │     └── LiveRunner (own API keys)
  │
  ├── Sub-Account: factory_winner_002
  │     ├── Play: rsi_bounce_21_4h
  │     ├── Allocated: $2,000
  │     └── LiveRunner (own API keys)
  │
  └── Sub-Account: factory_winner_003
        ├── Play: macd_cross_trend
        ├── Allocated: $1,500
        └── LiveRunner (own API keys)
```

## Resource Estimates

### Memory Budget

| Component | Per-Engine | 50 Engines | 500 Engines |
|-----------|-----------|------------|-------------|
| SimulatedExchange + Ledger | ~2 MB | 100 MB | 1 GB |
| Indicators (per TF) | ~5 MB | 250 MB | 2.5 GB |
| Structure state | ~3 MB | 150 MB | 1.5 GB |
| Candle buffers (3 TFs) | ~5 MB | 250 MB | 2.5 GB |
| **Total per engine** | **~15 MB** | **750 MB** | **7.5 GB** |

### CPU Budget

Each bar tick processes all engines. On exec TF = 15m, that's 1 tick
per 15 minutes — CPU is NOT the bottleneck for live sim. Even 500
engines processing 1 bar each takes < 1 second.

For backtesting (25,000 bars × 500 engines), CPU matters more.
ProcessPoolExecutor with 8 workers handles this.

### Network Budget

WebSocket data is shared — 1 connection per symbol regardless of engine count.
Bybit WebSocket costs ~5 KB/s per kline subscription. Even with 10 symbols ×
3 timeframes = 30 subscriptions → ~150 KB/s total. Negligible.

## Implementation Order

### Phase 1: LiveSimRunner (P0)

Build the core: LiveDataProvider + SimulatedExchange in a new runner.
Prove 1 engine works with real-time data and tracks PnL.

### Phase 2: DataFanout + MassRunner (P0)

Multiplex 1 WebSocket → N engines. Prove 50 engines run concurrently.

### Phase 3: StrategyGenerator (P1)

Template + param grid → N Play YAMLs. Integrate with mass backtest.

### Phase 4: Leaderboard + Scoring (P2)

Persistent tracking of live sim results. Auto-rank, auto-demote losers.

### Phase 5: Promotion + Sub-Accounts (P3)

Auto-promote winners to demo. Wire up Bybit sub-account management.

## Open Questions

1. **Indicator sharing** — If 50 engines on the same symbol all compute EMA(50),
   can we compute once and share? This would cut memory 50x for shared indicators.
   Tradeoff: coupling between engines, complexity.

2. **Database for results** — Backtest results currently go to artifact folders.
   At 1000s of plays, we need a queryable results store (SQLite? Postgres?).

3. **Genetic evolution** — The SYSTEM_VISION.md describes a genetic algorithm
   approach (DEAP + Ray). How does that integrate with the factory pipeline?
   Is it a separate stage or replaces the grid generation?

4. **Multi-symbol** — Current design is 1 symbol per engine. Some strategies
   trade multiple symbols. Defer or design for it now?

5. **State persistence** — If the factory process restarts, can live sim
   engines resume? SimulatedExchange state would need serialization.
